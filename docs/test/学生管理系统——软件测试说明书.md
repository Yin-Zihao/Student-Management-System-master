# 1 引言

## 1.1 编写目的

本文档用于明确学生管理系统的测试范围、测试策略、测试方法及验收标准，指导测试工作有序开展，验证系统功能、业务流程、数据准确性与安全性是否符合需求规格与概要设计要求，为系统验收提供依据。

## 1.2 项目背景

本系统为面向计算机学院的综合管理系统，采用前后端分离架构，实现学生管理、课程管理、成绩管理、科研管理、竞赛管理、党务管理、数据导入导出等功能，面向管理员、教师、学生三类角色使用。

## 1.3 参考资料

- 学管系统概要设计文档V1.2.5
- 学管系统需求规格说明书
- 软件工程课程设计相关规范

# 4 测试方法选取

本次测试选取以下标准软件测试方法，适用于单人执行、无需代码调试，符合课程设计要求：

1. **黑盒测试**  
   以需求与设计为依据，验证功能、输入输出、界面展示是否符合预期。

2. **功能测试**  
   对各模块功能点逐项验证，确保增删改查、统计计算、导入导出等可用。

3. **场景法测试**  
   模拟真实业务流程，按操作顺序执行，验证流程连贯性与数据一致性。

4. **权限测试**  
   验证不同角色（管理员、教师、学生）的功能访问范围与数据可见范围合法。

# 5 测试范围与测试点

## 5.1 登录模块

**测试方法：黑盒测试、功能测试、场景法测试**  
**测试步骤：**

1. 输入合法/不合法/为空的账号与密码，执行登录操作。
2. 使用不同角色账号登录，验证菜单与功能权限正确。
3. 执行退出操作，验证会话失效并返回登录界面。  
  **测试点：**
- 账号密码正确性校验
- 不同角色登录后功能展示正确
- 退出登录功能有效

---

3. 修改用户角色与状态，验证保存生效。

# 8 缺陷管理规范

## 8.1 缺陷提交规范

- 缺陷标题：模块名称+问题简述
- 重现步骤：按顺序描述操作流程
- 预期结果：需求规定的正确结果
- 实际结果：当前表现
- 附件：截图或日志

## 8.2 缺陷等级

1. 严重缺陷：功能不可用，影响核心流程
2. 一般缺陷：功能异常，可绕过但影响使用
3. 轻微缺陷：界面、文字、提示类问题

## 8.3 回归测试

开发修复后，测试人员重新执行对应用例，确认问题解决且不引入新缺陷。
